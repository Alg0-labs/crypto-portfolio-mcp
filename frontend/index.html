Get started

Connect your AI assistant

Works with any MCP client. Pick yours, paste the config, and you're talking to live crypto data in under a minute.

1

Grab a free Moralis API key at moralis.io — needed only for the four wallet tools. Market-data tools need no key at all.

2

Add the config below to your client, replacing YOUR_MORALIS_KEY with your own key.

3

Restart the client and ask: "What's the price of ETH?" or "Analyze wallet 0x… on Ethereum."

{
  "mcpServers": {
    "coinlens": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "__MCP_URL__",
        "--header",
        "X-Moralis-Key:YOUR_MORALIS_KEY"
      ]
    }
  }
}

Add to claude_desktop_config.json via Settings ▸ Developer ▸ Edit Config. The mcp-remote bridge ships with npx; nothing to install.

{
  "mcpServers": {
    "coinlens": {
      "url": "__MCP_URL__",
      "headers": {
        "X-Moralis-Key": "YOUR_MORALIS_KEY"
      }
    }
  }
}

For clients that support remote MCP servers with custom headers directly.

MCP endpoint   __MCP_URL__
Transport      Streamable HTTP (stateless)
Auth header    X-Moralis-Key: YOUR_MORALIS_KEY   (portfolio tools only)
Health check   __HEALTH_URL__

We store no keys. Every request carries the caller's own key, so usage always stays on your account.

Design principles

  • Protocol-native. The server is a first-class MCP server, not a REST API with an adapter. Tools are declared through @server.list_tools() and dispatched through @server.call_tool(), so any compliant client discovers and calls them with zero custom glue.
  • Two transports, one codebase. Locally it runs over stdio (the classic Claude Desktop integration). Hosted, the same server is wrapped in a Starlette ASGI app exposing the Streamable HTTP transport at /mcp. The transport is selected at boot by the MCP_TRANSPORT env var.
  • Stateless & multi-user. The HTTP transport runs in stateless mode — every request is fully independent, with no server-side session affinity. That makes it safe to host publicly and trivial to scale horizontally.
  • Bring-your-own-key. The server holds no Moralis key. Each caller sends their own key per request in the X-Moralis-Key header, so the operator never pays for users' usage and there is no shared secret to leak or abuse.

Components

  • MCP core (src/server.py) — tool schema declarations, the dispatch table, and the 12 tool handlers. Also builds the Starlette app (build_http_app()) and the stdio entrypoint.
  • API clients (src/api/) — CoinMarketCapClient for market data and MoralisClient for on-chain wallet data. Both are async httpx clients with timeouts and clean shutdown wired into the app lifespan.
  • Portfolio engine (src/utils/portfolio.py) — turns raw balances into allocation, valuation, risk signals and rebalancing recommendations.
  • Domain utilities (src/utils/) — chains.py (the eight-network registry), validators.py (address & symbol checks), and formatters.py (human-readable output the model relays back to you).

Request lifecycle

A single tool call — say "analyze wallet 0x… on Ethereum" — flows through these stages:

1The model decides a tool is needed and emits a JSON-RPC tools/call request to the client.
2The client (or the mcp-remote bridge) POSTs it over HTTPS to /mcp, carrying the X-Moralis-Key header.
3The stateless session manager parses the request and routes it to the matching tool handler.
4The handler validates arguments, then calls CoinMarketCap and/or Moralis with the caller's key.
5The portfolio engine and formatters shape the response into readable text content.
6The result is returned as an MCP TextContent payload; the model reads it and answers you in natural language.

Deployment

  • Container. A single Dockerfile builds the server; MCP_TRANSPORT=http selects the hosted transport and the app binds to the platform-injected PORT.
  • Render blueprint. render.yaml deploys the container with a /health check and no secret env vars — the BYO-key model means there's nothing to configure per user. Any Docker host (Railway, Fly.io, a VPS) works identically.
  • Separation of concerns. The landing page is a static site (hosted on Vercel, so it loads instantly), while the MCP backend runs as a persistent container on Render. The two are decoupled: the page only talks to the backend when a client actually connects.
